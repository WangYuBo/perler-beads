# 拼豆项目飞书协作规范

## 1. 文档目的
统一需求提出与开发流程，减少沟通成本，确保需求可追溯、开发有依据。

---

## 2. 角色分工

| 角色 | 职责 |
|------|------|
| **需求方**（同事们） | 提出功能/优化/Bug 需求，补充细节，验收结果 |
| **开发方**（你） | 评估可行性，执行开发，反馈进度，交付代码 |

---

## 3. 需求提交规范

### 3.1 新建需求格式
每个需求单独一行/一段，使用以下模板：

```
【优先级】需求标题
- 背景/问题：为什么要做这个？
- 具体描述：期望的效果是什么？
- 验收标准：怎么算完成？
- 参考/截图：（可选）贴图或链接
```

### 3.2 优先级标签

| 标签 | 含义 | 响应时间 |
|------|------|----------|
| 🔴 **P0-紧急** | 阻断性 Bug，核心功能不可用 | 24h 内 |
| 🟠 **P1-高优** | 重要功能缺失，影响体验 | 3 天内响应 |
| 🟡 **P2-中优** | 体验优化，有替代方案 | 1 周内响应 |
| 🟢 **P3-低优** | 锦上添花，可排期 | 酌情处理 |

### 3.3 状态标签（开发方维护）

| 标签 | 含义 |
|------|------|
| ⏳ **待评估** | 刚收到，未开始分析 |
| ✅ **已确认** | 确认可行，进入排期 |
| 🚧 **开发中** | 正在编码 |
| 🧪 **待测试** | 代码完成，可验收 |
| ✔️ **已完成** | 已上线或合并 |
| ❌ **暂不做** | 经评估暂不实现，需说明原因 |
| ❓ **需补充** | 信息不足，需要需求方补充 |

---

## 4. 沟通规则

### 4.1 需求方
- 一个需求点只写一次，不要分散在多处
- 改需求时**不要删除原文**，用「删除线」划掉+补充新内容
- 贴图优先用飞书截图，避免外链失效
- 批量需求可分条列出，但一次不宜超过 5 条

### 4.2 开发方
- 每条需求至少给一个状态反馈（⏳/✅/❌/❓）
- 技术评估意见直接批注在原需求下方
- 实现有困难时，提出替代方案而非直接拒绝

---

## 5. 文档结构建议

飞书文档可按以下分块：

```
📋 待处理需求（按优先级排序）
   └─ 新提出的、待评估的需求

🚧 开发中
   └─ 已确认、正在实现的需求

✔️ 已交付（按日期倒序）
   └─ 已完成的需求，可快速回顾

📝 需求池
   └─ P3 低优、暂不做的需求，不丢失

💬 讨论记录
   └─ 复杂需求的决策过程
```

---

## 6. 快速示例

**需求方提交：**

> 🟡 **P2-中优**：颜色统计面板增加"按数量排序"功能
> - 背景：目前颜色按色号排序，找用量大的颜色不方便
> - 描述：在颜色统计区增加排序切换按钮（按色号/按数量）
> - 验收：点击按钮后颜色列表按数量从多到少排列

**开发方响应：**

> ✅ **已确认** | 预计本周五完成
> - 技术方案：在 ColorStats 组件增加 sortBy 状态，用 useMemo 排序
> - 无技术风险
